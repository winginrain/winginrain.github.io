uml核心元素——think in UML读书笔记2

Posted on 2019-04-24 | Edited on 2019-05-20 | In Thinking in UML

UML版型（Stereotype）

UML中每个元模型都有很多版型。e.g. 用例：业务用例，业务用例实现；类：接口、边界类、实体类；

参与者

基本概念

参与者（actor）在建模过程中处于核心地位，是在系统之外与系统交互的某人或某事物。
参与者特性：

  1. 位于系统之外。主动启动这个业务。
  2. 参与者可以非人。只要主动启动的一方即可认为是成参与者。

参与者相关概念

业务主角：参与者的一个版型，特别用于定义业务的参与者在需求阶段使用。业务主角是业务系统有着交互的人和事物，他们用于确定业务范围。业务主角针对的是业务人员而非计算机用户，由业务主角的业务模型建立的计算机系统模型。
业务工人：系统边界的一部分，被动参与业务，没有具体的目的。（业务主角：购票人；业务工人：人工订票）
涉众（stakeholder）：干系人，与这个系统有利益关系的一切人和事。参与者是涉众代表。参与者对系统的要求直接影响系统的建设，他们的要求就是系统需求的来源。
用户：系统的使用者，参与者的代表，实例或代理。
角色：系统中一类职责。一个用户可以代理多个参与者，被指定多个角色。

用例

用例基本概念

用例：把现实世界的需求捕获下来的方法。用例定义了一组用例实例，其中每个实例都是系统所执行的一系列操作，这些操作生成特定主角可以观测的值。用例=交互+可观测的有意义的结果+活动。
完整的用例定义=参与者+前置条件+场景+后置条件
系统的功能性：对系统有需求的参与者要做的一些事情构成的。所有需求可以通过用例达到，那么即可确定该系统。

用例的特征

  1. 用例是相对独立的。不需要与其他用例交互而独立完成参与者的目的。
  2. 用例的执行结果对参与者来说是可观测的和有意义的。
  3. 用例必须由一个参与者发起。
  4. 用例必然以动宾短语形式出现。
  5. 一个用例就是一个需求单元，分析单元，设计单元，开发单元，测试单元
  6. 用例的来源是参与者对系统的期望

    用例的粒度

    用例的粒度——以多大的事件作为一个完整的用例。在概念建模阶段，以每个用例能描述一个完整的事件流为宜。在系统建模阶段，用例视角是针对计算机的，用例的粒度用一个用例能够描述操作者与计算机的一次完整交互为宜。
    用例的粒度选择在不同系统中有较大差异，一般用于适应不同的需求范围。一般在10-50之间。

    业务用例（business use case)

    用例版型的一种，用于需求阶段的业务建模。一个业务用例的多个业务用例实现是为了达到同一个目的。业务建模是针对客户业务的模型，与计算机系统建模无关，是业务领域的一个模型。
    业务用例有多种实现方式，同一个接口可以有多个实现类，业务用例实现是实现对象追溯到需求的一个重要环节。我们通过业务用例实现将得到关键业务对象，再从业务对象转化到设计对象。

    概念用例

    概念用例用于获取业务模型中关键概念，分析出业务模型组的核心业务结果以得到一个易于理解的业务框架。业务架构在这个阶段产生。
    作为概念模型中的核心元素，概念用例用于获取业务用例中的核心业务逻辑。概念用例是从业务用例到系统用例过渡是重要的指导。

    系统用例

    系统用例可简称为用例，是用于定义系统范围、获取功能性需求的，是软件系统开发的全部范围，是我们得到的最终需求。
    （系统）用例实现是实现对象追溯到需求的一个重要环节。做完用例模型后，直接进入数据库表设计以及类设计。

边界

边界在UML图符中的定义是一个矩形框。在面向对象中，每个对象都有一个边界，外界只能通过边界来认识对象，操作对象，但无法进入对象内部。
边界有以下性质：

  1. 边界决定视界。通过变幻边界，改变世界，从多个侧面描述同一个信息，从而最大程度地附和真实的需求。
  2. 边界决定抽象层次。可以通过自顶向下方法（通过逐步缩小边界，进而影响我们观察到的事物，决定相应的抽象层次，使得分析粒度逐步细化），也可以自底向上方法（边界设定到较小范围，再逐步扩大）
  3. 灵活使用边界。边界可以在需求方面以及设计层面发挥重要作用。对于软件设计，有很大的信息量，既要实现需求，又要保证性能，有扩展能力，且友好易用——可以通过设计适当的边界。

业务实体

业务实体是类(class)的一个版型，用于在业务建模阶段建立领域模型。业务实体抽象出问题领域内核心和关键的概念。官方定义：业务实体代表业务角色执行业务用例式所处理或使用的事物，经常代表某个对多个业务用例或用例实例有价值的事物。
业务实体的性质如下：

  1. 业务实体来自现实世界
  2. 业务实体在分析业务流程（即业务用例场景）的过程中发现的。即业务实体必须至少被一个业务用例场景使用或创建。
  3. 业务实体是类的一个版型，具有对象的所有性质，包括属性，方法，对象的独立性。对象的独立性是指业务实体只应当包含它自身固有的特性，而不应包含外界是如何使用它的信息。

    业务实体的属性以及方法

    属性是用来保存业务实体特征的一个记录，业务实体的属性集合决定了它的唯一性。
    方法是访问一个业务实体的句柄，规定了外部可以怎么来使用它。

    获取业务实体

    获取业务实体的方法：
  4. 建立业务用例场景。业务用例场景时参与者实现其业务目标的过程描述。
  5. 从业务用例场景中诸葛分析动词后面的名词。从业务实体的备选对象选取初始的业务实体。
  6. 分析业务实体之间的关系。

包

包是一种容器，可以将信息分类，从而形成逻辑单元。目的： 整合复杂信息，对语义上相关或某方面具有共同点的信息进行分包。
包是UML中常用的元素，用于容纳并为其他元素分类。包可以容纳任何UML元素（用例，业务实体，类图）。好的分包应该具有高内聚，低耦合的性质。

分包原则：

  1. 元素分包后，同一个包中的元素应相互联系紧密。同时这些元素具有某些相同的性质，使得包可以抽象出一些接口代表包内事物与包外的事物交互。
  2. 修改一个包中元素，其他包中的内容不受影响。包之间无依赖关系或松耦合关系。
  3. 若无法解除依赖关系，那么包之间的依赖关系不会被传递
  4. 报纸剪的依赖应该是单向的。

    常用的包的版型

  5. 领域包（Domain Package）：用于分类业务领域内的业务单位，领域包视图可用于展示这些业务领域的高层次关系。
  6. 子系统（Subsystem）：用于分类系统内的逻辑对象并形成子系统。子系统包视图可用于展示系统的高层次逻辑结构关系。
  7. 组织结构（Orgnaizatioin unit）：组织结构包用于分类业务领域中的组织结构，用于表述企业的组织结构。
  8. 层（Layer）：分类软件中的层次，用于展示软件的架构信息。

    分析类

    分析类：用于获取系统中主要的职责簇，可代表系统的原型类。
    分析类是从业务需求向系统设计转化过程中最为主要的元素，在高层次抽象出系统实现业务需求的原型，业务需求通过分析类逻辑化。
    分析类由边界类，控制类，实体类三部分组成，都是类的版型。

    边界类

    边界类：用于对系统外部环境与其内部运作之间的交互进行建模的类。交互包括：转换事件，记录系统表示方式中的变更。任何有交互的关键对象之间都应考虑建立边界类。

    边界类的常用场景：

  9. 参与者与用例之间应当建立边界类。
  10. 用例与用例之间如果有交互，应当建立边界类。（可演化为一组API、一组JMS消息或一组代理类）
  11. 用例与系统边界之外的非人对象。
  12. 在相关联的业务对象有明星的独立性要求。

    边界类的特点：

  13. 边界类应该有助于提高系统的可用性
  14. 边界类应该尽可能地保持在较高的层次上
  15. 边界类应该合理封装介于系统与角色之间的交互
  16. 主角改变为系统提供输入的方式，只需改变边界类
  17. 系统改变输出的方式，只需改变边界类
  18. 边界类必须知道其他对象类型的需求，并且相对于系统内部元素保持其可用性和有效性。

    控制类

    控制类用于对一个或几个用例所特有的控制行为进行建模。控制对象（控制类实例）通常控制其他对象。
    在UML的定义中，控制类主要起到协调对象的作用。比如边界类通过控制类访问实体类，或实体类通过控制类访问另一个实体类。
    在设计阶段，控制类可以被设计为Session Bean、COM+、Server Let、Java类、C++类等。

    实体类

    实体类用于对必须存储的信息和相关行为建模的类。实体对象（实体类的实例）用于保存和更新一些现象的有关信息。实体类通畅时候永久性的，它们所具有的属性和关系是长期需要的，甚至在系统的整个生存期都需要。
    在设计阶段，实体类可以被设计为Entity Bean、POJO、SDO、XML Bean等设计类甚至是一条SQL语句。在架构角度上，实体类主要位于数据持久层，实体类的获取对架构设计中的数据持久层有指导意义。

    分析类的作用

    分析类是从业务需求向系统设计转化过程中最为主要的元素，在高层次抽象出系统实现也无需求的原型，业务需求通过份额锡类被逻辑化，成为可以被计算机理解的语义。
    分析类应高于设计实现、高于语言实现、高于实现方式，专注于解决需求问题。

设计类

设计类是系统实施中一个或多个对象的抽象。设计类所对应的嗯对象取决于实施语言，设计类用于设计模型中，直接使用与编程语言相通的语言来描述。
分析类：为设计类中所需要的界面、逻辑和数据提供抽象基础，设计类可以从分析类中演化出来。

类

类对对象进行定义，而对象又实现用例。类的来源可以是用例实现对系统所需对象的需求（实现业务需求），可以是任何以前已开发的对象模型。类说明对象是什么，也决定对象拥有什么属性、方法。
类是对对象某一方面特征的归纳和抽象，而对象则是类实例化的结果。

属性

属性是对象特征，表明对象的唯一性。只有对象单独具有的特征才建模为属性，否则使用类的关联关系或聚合关系对特征进行建模。

方法

访问对象或影响其他对象的属性或关系的唯一途径是方法。对象的方法由它的类进行定义。 类定义的方法是由实例化的对象执行的，也可以由类执行（JAVA中的静态方法）

可见性

  1. 公有：出了类本身以外，属性和方法对其他模型元素是可视的，应尽量少用
  2. 保护：属性和方法只对类本身、它的自雷或有缘（取决于具体语言）是可视的。（默认）
  3. 私有：属性和方法只对类本身和类的友元是可视的。可以用在不希望子类继承属性和方法的情况下。它提供了从超累对子类去耦的方法，并且减少了删除或排除未使用继承操作的需要。
  4. 实施：属性和方法只在类本身的内部是可视的。

    关系

    抽象出对象之间的联系，让对象构成某个特定的结构。

    关联关系（association）

    关联关系是用一条直线表示。用于描述不同类的对象之间的结构关系。在一段时间内将多个类的实例链接在一起。可以使用关联关系表示一个对象了解其他对象。
    关联=知道
    $A - B$ ：相互知道
    $A \to B$ ：单向知道，A知道B，B不知道A

依赖关系（dependency）

依赖关系是用一条带箭头的虚线表示的。e.g. $A \dashrightarrow B$ （A依赖于B）。
依赖关系描述一个对象的修改会导致另一个对象的修改这样的关系。依赖=知道+使用
e.g. A对象保存B对象的ID，但没有对B对象操作，仅仅知道 ——使用关联
A对象使用B对象的属性或方法，B的修改会导致A的修改——使用依赖

扩展关系（extends）

扩展关系是用一条带箭头的虚线加版型<\>来表示的。
扩展关系用于在用例模型中说明想基本用例中某个扩展点插入扩展用例。扩展用例带有抽象性质的，表示用例场景中的某个支流，由特定的扩展点触发而被启动。扩展用例应当用在概念用例模型中，通过分析业务用例场景抽象出关键的可选核心业务而形成扩展用例。
在以下理由下可以使用扩展关系：

  1. 表明用例的某一部分是可选的系统行为
  2. 只在特定条件下才执行分支流（如触发警报）
  3. 可能有一组行为段，其中一个或多个段可以在基本用例的扩展点出插入。所插入的行为段取决于执行基本用例时与主角进行的交互。

  4. 多个基本用例都有可能触发一个可选的分支流。

    e.g. 打电话过程中，收到新到的电话——选择保留当前通话

    包含关系（include）

    包含关系是用一条带箭头的虚线加版型<\>表示，用于用例模型，说明基本用例的用例实例过程中插入的行为段。
    包含用例总是带有抽象性质的，基本用例可控制与包含用例的关系，并可依赖于执行包含用例所得的结果，但基本用例和包含用例都不能访问对方的属性。

实现关系（realize）

实现关系是用一条带空心箭头的虚线表示。用于在用例模型中连接用例和用例实现，说明基本用例的一个实现方式。实现所代表的含义是，基本用例描述了一个业务目标，但是该业务目标有多重可能的实现途径，每一种实现途径可以用用例实现来表示。用例实现与基本用例之间就构成实现关系。

精化关系（refine）

精化关系是用一条带箭头的虚线加版型<\>表示，特别用于用例模型，一个基本用例可以分解出许多更小的关键精化用例，这些精化用例更细致地展示基本用例的核心业务。
精化关系表示由基本对象可以分解为更明确、精细的子对象，这些子对象并没有增加、减少、改变基本对象的行为和属性，只是更加细致和明确化了。精化关系仅仅由于建模阶段。

泛化关系（generalization）

泛化关系是用一条带空心箭头的直线表示。泛化关系可用于建模的任意阶段，说明对象之间的继承关系。在泛化关系中，基本对象被泛化后成为子对象后，子对象继承了基本对象的所有特征，并且子对象可以增加、改变基本对象的行为和属性。
泛化关系表示一个类对另一个类的继承（后代、祖先）。
下图表示A继承自B

聚合关系（aggregatioin）

聚合关系是用一条带空心菱形箭头的直线表示。
聚合关系用于类图，特别用于表示实体对象之间的关系，表述整体由部分构成的语义。
其中，聚合关系中整体和部分不是强依赖的。整体不存在是，部分依然存在。例如部门撤销后，人员依旧存在。

组合关系（composition）

组合关系是用一条带实心菱形箭头的直线表示。组合关系用于类图，特别用于表示实体对象个关系，表达整体拥有部分语义。组合关系是一种强依赖的特殊聚合关系，如果整体不存在了，则部分将不存在。例如母公司解体了，子公司将不存在。

组件

组件是系统中实际存在的可更换部分，实现特定的功能，符合一套接口标准并实现一组接口。组件代表系统中的一部分物理实施，包括软件代码或其等价物。
建模过程中，我们通过组件这一元素对分析设计过程中的类/接口等进行逻辑分类，一个组件表达软件的一组功能。出于构件化的需要，把那些紧密合作的类和接口组合起来实现一组特定的功能。一个类可能被分派给多个组件以完成该组建的功能。当组件被编译或打包成一个物理文件，每个组件都拥有这个类的一个拷贝或引用该类的途径。
在UML中组件定义为任何的逻辑代码模块。即任意几个类一组合就可以成为一个组件。组件之间唯一的关系是依赖。
该书作者认为，组件应该具有完备性、独立性、逻辑性和透明性。一个组件应当是一个独立的业务模块，由完备的功能，可独立部署，一个组件可以看成是一个完备的服务。
在SOA架构的观点上看，一个SOA服务与其他服务是没有依赖关系的，服与服务之间仅仅保持着松耦合的通信关系。松耦合是指在发出消息前，组件之间甚至不知道对方的存在，之间仅仅通过架构沟通。

组件特点

  1. 完备性：
    组件包含一些类和借口哦，一个组件应当能够完成一项或一组特定的业务目标。不需要调用多个组件来完成一个业务请求。
  2. 独立性：
    组件应当是可以独立部署的，与其他组件无依赖关系，最多保持关联关系。
  3. 逻辑性：
    组件是从软件构件设计的观点来定义的，并非从需求中可以直接导出。组件的定义是为了鬼话系统结构，将一个复杂的系统分解为一个个具有完备功能的，可独立部署的小模块。
  4. 透明性：
    组件的修改应当只涉及组件的定义以及组件所包含的类的重新指定，而不应当导致类的修改。即你可以换类，但你不能修改类本身。

    组件的用途

    组件是从系统结构的角度来划分分析设计的结果。组件在以下场景中较有用：
    1. 分布式应用：系统的功能可能被部署在异构环境下，一个业务目标可能需要经历两个甚至多个节点才能完成。此时可以规划组件，从而将各个组件分别部署在不同节点上。
    2. 应用集成：遗留系统与新系统的结合，可以使用组件
    3. 第三方系统：将系统中提供给第三方系统使用的功能定义成一系列组件，让第三方通过组件访问本系统。
    4. SOA服务（Service Oriented Architecture）：将系统结构划分成粗粒度的服务组件SCA，每个服务组件遵循一系列的标准和规范，通过标准的同性协议与其他服务交互，服务与服务之间是松耦合的。

节点

节点是带有至少一个服务器、内存以及可能带有其他设备的处理元素。e.g. 服务器、工作站、客户机。节点是应用程序的部署单元。节点元素特别用于部署视图，描述应用程序在物理结构是如何部署在应用环境中的，是一种包括软、硬件环境在内的拓扑结构描述。
可在分布式应用环境以及多设备应用环境中使用。
